Solution Overview

Design Principle: The platform enriches (OOB business rule + Nexthink sim). The AI agent decides (custom 4-phase system prompt + 6 tools). Humans validate. When a user escalates from a chatbot, ServiceNow autonomously triages the incident before any analyst opens it.

Enrich (Business Rule + Nexthink)

On incident creation a business rule fires the NexthinkSimulator Script Include — returning real-fidelity device telemetry (CPU, memory, disk, network, crash history, uptime). Diagnostics are written to the incident as structured work notes before the AI agent begins.

Triage (AI Agent Studio)

A Zurich-native AI agent (sn_aia_* tables) with a 4-phase system prompt autonomously reads enrichment data, generates the short description, recommends category / subcategory / CI, and routes to the correct group — all via 6 tool calls.

Validate (Human-in-the-Loop)

The fully triaged incident lands in Service Operations Workspace with a complete AI work note. The analyst confirms, adjusts if needed, and closes. The Analytics dashboard tracks AI accuracy and MTTR improvement over time.

WHY

The Value of Two Workflows

Chatbot-escalated incidents arrive with no device context. WF1 adds the missing telemetry. WF2 reasons over it. Together they eliminate manual investigation and routing before the analyst opens the ticket.

WF 1

Nexthink Enrichment Pipeline

Business rule fires on incident insert. Calls NexthinkSimulator Script Include with the caller's device CI. Parses the JSON response. Writes structured diagnostics (CPU, memory, disk, network, crashes) to incident work notes before the AI agent reads anything.

WF 2

AI Triage Agent (sn_aia_agent)

sn_aia trigger fires after enrichment. Agent worker reads the enriched incident (T-01), categorizes via sys_choice lookups (T-02, T-03), links the device CI (T-04), searches KB (T-05), and writes a structured triage work note with full reasoning (T-06).

WHAT

Sequential Handoff

WF1 and WF2 are fully sequential — enrichment writes data, agent reads data. The agent's T-01 (Get Incident Details) reads the work notes WF1 wrote. No shared state other than the incident record. In production, swap the Script Include for a real Nexthink REST spoke — WF2 is unchanged.

Data Model

Zero custom schema. No custom fields on incident. No custom tables. The AI work note is written to work_notes — a standard OOB field. Device diagnostics are also work notes. Every triage artifact is traceable through the OOB incident timeline.

AI Triage Work Note — what the agent writes

NEXTHINK TRIAGE SUMMARY — Phase 1-4 complete ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ DEVICE HEALTH (from Nexthink): CPU usage: 94% ⚠ HIGH Memory: 87% ⚠ HIGH Disk free: 8% ✗ CRITICAL Crashes (7d): 3 ⚠ ELEVATED Network: Good ✓ ASSESSMENT: Device is under severe resource pressure. High CPU + low disk are consistent with the user's reported freezing behavior. Assignment Routing Architecture

Defense-in-depth routing. The AI agent recommends the category. OOB assignment rules enforce the group. Even if the agent miscategorizes, the rule prevents misrouting. Two independent systems, one correct outcome.

Routing Logic

// AI agent sets category (via tool T-02/T-03 recommendation) // OOB assignment rule engine reads category and enforces group IF incident.category = "Hardware" → assignment_group = Desktop Support ELIF incident.category = "Network" → assignment_group = Network Operations ELIF incident.category = "Software" → assignment_group = Application Support ELSE → assignment_group = Service Desk (fallback) // The AI work note also records the recommended group as a talking point. // The OOB rule is the authoritative assignment — not the AI recommendation alone.

AI Agent Configuration

The system prompt explicitly maps category to group in Phase 4. The agent writes its routing recommendation to the work note with reasoning ("CPU 94% + disk 8% = Hardware/Performance"). The OOB rule then confirms the routing independently.

Post-POC Path

For live deployments: replace the category-to-group mapping with a reference table keyed to CMDB CI assignment groups (matching production support ownership). No agent changes required — only the assignment rule conditions change.
